# Membership & Badges

The smart-contract powering Tansu keeps track of community members **on-chain** and assigns them _badges_. These badges are more than simple labels – they carry a **numerical weight** that is used for weighted voting in the default governance mode.

## Registering as a member

Anyone can register themselves on-chain by calling:

```bash
stellar contract invoke \
  --source-account G... \
  --network testnet \
  --id <TANSU_CONTRACT_ID> \
  -- \
  add_member \
  --member_address G... \
  --meta <ipfs_cid>
```

`meta` is optional. When supplied it must be an **IPFS CID** pointing to a folder that contains:

- `profile.json` – free-form JSON metadata (name, description, …)
- `profile-image.png` – avatar (optional)

The dApp will automatically render this information in the **Member profile** modal.

## Badges

Maintainers can assign one or several badges to a member for a given project. Badges are stored on-chain and influence the maximum weight a member's vote can carry in **badge-based** proposals.

| Badge     | Weight     | Typical use-case                |
| --------- | ---------- | ------------------------------- |
| Developer | 10 000 000 | Core code contributors          |
| Triage    | 5 000 000  | Issue triage & review           |
| Community | 1 000 000  | Active community member         |
| Verified  | 500 000    | A member which has been KYC/KYB |
| Default   | 1          | Anyone else                     |

> The weight column reflects the numeric values found in the `Badge` enum of the contract (see source code). When several badges are present their weights are **summed** to obtain the member's total voting power.

### Adding badges

Only maintainers of a project can add badges:

```bash
stellar contract invoke \
  --source-account <MAINTAINER_G...> \
  --network testnet \
  --id <TANSU_CONTRACT_ID> \
  -- \
  set_badges \
  --maintainer <MAINTAINER_G...> \
  --key <project_key> \
  --member <member_G...> \
  --badges '[Developer,Community]'
```

The dApp exposes a convenient **"Add badge"** modal that wraps this call.

## Voting weight modes

Effective vote weight depends on the proposal type and project configuration. The contract's `get_max_weight` helper applies to badge-based and NQG-weighted projects; token-based proposals use a different path at vote time.

| Mode | When it applies | How weight is determined |
| ---- | --------------- | ---------------------- |
| **Badge-based** | Default; `create_proposal` with `token_contract: None` on a normal project | Sum of assigned badge enum values (default **1** if unregistered or no badges) |
| **Token-based** | `create_proposal` passes a `token_contract` | Voter chooses `weight` in whole token units (locked as collateral); badge cap is skipped |
| **NQG contract** | Project key matches admin-configured `NqgProjectKey` (SCF Public Goods deployment) | `get_max_weight` calls the external NQG contract; returns **0** if score ≤ 4M (pilots only) |

### NQG contract weight

One designated project — the Stellar Community Fund Public Goods Award deployment — uses live Neural Quorum Governance scores instead of badges. The Tansu admin configures the NQG contract via `set_nqg_contract`. Voters below the 4M score threshold receive zero weight (pilots only).

### Token-based proposals

When a proposal is created with a `token_contract`, voting weight equals the amount of tokens the voter locks as collateral, not `get_max_weight`. See [Governance & Proposals](./governance.mdx) for proposal creation and vote validation.

## Query helpers

Smart-contract helpers make it easy to retrieve information:

- `get_member` → full `Member` structure (projects + badges + meta)
- `get_badges` → badge holders for a project
- `get_max_weight` → badge sum or NQG score depending on project; token proposals bypass this at vote time

These endpoints are all used by the dApp and are public – anyone can call them from an RPC or another contract.
